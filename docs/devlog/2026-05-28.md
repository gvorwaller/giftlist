# 2026-05-28 Devlog

## 11:57 EST - Fix: "tomorrow" label off by one day after 8 PM EDT (td-85b243)

### Summary
The relative date label ("tomorrow", "in 2 days") on occasion cards was off by one day when viewed after 8 PM EDT, because the server derived "today" from its UTC clock instead of the user's Eastern timezone.

### What Was Done

- Root cause: `todayMidnightUTC()` used `new Date().getDate()` to extract the calendar date, but the DO production server runs in UTC. After midnight UTC (8 PM EDT), the server's "today" rolled forward a day before the user's day changed — making `daysUntil` one too few.
- Example: on Wednesday 5/27 evening, Dagny's graduation on Friday 5/29 showed "tomorrow" (daysUntil=1) instead of "in 2 days" (daysUntil=2). The formatted date "Friday, May 29" was correct because it's rendered client-side in the browser's local timezone.
- Fix: `todayMidnightUTC()` now uses `Intl.DateTimeFormat` with `timeZone: 'America/New_York'` to extract the calendar date. The formatter is allocated once at module scope (no per-call overhead). Both app users are in Eastern time.
- Affects all three pages that compute `daysUntil`: Today (`/app/today`), People detail (`/app/people/[id]`), and People list (`/app/people`).

### Files Modified
- `src/lib/server/occasions.ts` — `todayMidnightUTC()` now derives date from Eastern time

### Key Technical Insight
On a UTC server, `new Date().getDate()` returns the UTC calendar date, which diverges from the US Eastern date after 8 PM EDT. The noon-UTC anchoring (td-68e82a) correctly prevents the *serialized* date from shifting a day in the browser, but the *comparison anchor* for `daysUntil` also needs timezone awareness. `Intl.DateTimeFormat.formatToParts()` is the clean way to extract date components in a specific timezone without string-parsing hacks.

### Next Steps
- If the app ever supports users in other timezones, `America/New_York` would need to become a per-user or per-request setting.
