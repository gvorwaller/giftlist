# 2026-05-11 Devlog

## 09:19 EDT — Amazon tracking bridge idea captured for later

### Summary
Discussed a bounded follow-on idea for Amazon packages whose shipped emails do
not expose the underlying `TBA...` tracking ID directly. Instead of attempting
full Amazon automation, captured a staged "tracking bridge" concept: store the
email's Amazon tracking link on the package, let the user open that link on
mobile, optionally paste the `TBA...` into the app, and later support an
Amazon-specific refresh path that writes status back into the existing tracking
fields without involving Shippo.

### What Was Decided
- **Do not overbuild this now**: Shippo is not the right backend for Amazon
  Logistics tracking, so the right model is a separate Amazon refresh path, not
  trying to force `TBA...` through existing Shippo registration.
- **Best low-friction MVP**:
  - parse/store the Amazon "Track package" URL from shipped/delivered emails
  - show an `Open Amazon tracking` button on package detail
  - optionally support manual paste of `TBA...` into the existing tracking
    field
- **Later enhancement only if worth it**: a helper could attempt to extract the
  `TBA...` from the authenticated Amazon page, but that is explicitly phase 2+,
  not the starting point.

### Artifacts Created
- Sketch doc: `docs/2026-05-11_amazon-tracking-bridge-sketch.md`
- td task: `td-b221ae` — "Amazon tracking bridge from email link to TBA refresh"

### Key Technical Insight
**Manual TBA paste is a viable first step, but it must branch away from
Shippo.** Once a package has `tracking_number = TBA...`, the app can still use
its existing tracking display fields (`tracking_status`,
`tracking_status_at`, `tracking_estimated_delivery`) while sourcing those
values from an Amazon-specific fetch path rather than the current carrier/Shippo
pipeline.

### Next Steps
- Leave `td-b221ae` open as a later-phase feature.
- If this becomes worth doing, start with schema + parser + package-detail link
  work before any helper extraction automation.

### Key Technical Insight
**The Codex spec said "save URL + manual paste + scrape Amazon for status." The biggest spec mod I made was to parse the URL from `bodyHtml` instead of `bodyText`.** The current Amazon parser reads `msg.bodyText` exclusively (line 198), and Gmail's text alternative often loses anchor `href` attributes — the user-facing "Track package" link is an HTML button. `GmailMessageFull` already exposes `bodyHtml` (gmail-reader.ts:19), so this was free. Without that change, the parser would have extracted nothing on a majority of real-world emails.

The second meaningful change was **routing by TBA-shape OR shipper-identity**, not by the free-text `carrier` column. Historically the `carrier` text gets populated by both the parser ("Amazon Logistics" if the parser saw that phrase) and Shippo (whatever Shippo decides). Two signals can disagree on the same gift. Tracking-number shape is the only authoritative source for "is this Amazon Logistics"; shipper identity is the only authoritative manual override. Free-text carrier is downstream noise — bypassing it makes the branch deterministic.

The Phase 3 scraper is honestly the most fragile part of the card. `track.amazon.com` is a SPA whose static HTML rarely carries status text — the JSON probe is the realistic path, and even that's undocumented and shape-varying. The whole module returns all-null and never throws, so when it inevitably breaks, the UI degrades to the deep-link button (which always works) and the user manually checks. That's the right failure mode for a non-critical feature.
