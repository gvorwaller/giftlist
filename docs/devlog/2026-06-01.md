# 2026-06-01 Devlog

## 11:58 EST - Fix Today status stuck on completed occasion (td-2de362)

### Summary
Ambree Songoninni's upcoming birthday showed "GIFT HANDLED" on the Today page because a completed gift from her past graduation occasion leaked into the status computation. Fixed two query paths in `annotateGiftState()`.

### What Was Done
- Diagnosed two vectors in `annotateGiftState()` that allowed stale gift statuses to bleed onto future occasions:
  1. `exactStmt` queried `(person_id, occasion_id)` without filtering by `occasion_year` — last year's "given" gift on a recurring occasion (birthday) matched this year's occurrence
  2. `looseStmt` claimed NULL-occasion gifts (Amazon imports) without excluding handled statuses — a completed unassigned gift got attributed to the nearest future occasion
- Added `AND (occasion_year IS NULL OR occasion_year = ?)` to `exactStmt` so only current-year (or unscoped) gifts match
- Added `AND status NOT IN ('given', 'returned')` to `looseStmt` so completed gifts aren't claimed by future occasions
- Deployed to production, health check green

### Files Modified
- `src/lib/server/today.ts` — two SQL query edits + call site update in `annotateGiftState()`

### Key Technical Insight
The `gifts` table stores `occasion_year` and it's reliably populated by both manual entry and Amazon/tracking imports, but the Today page annotation queries never used it. The `OR occasion_year IS NULL` fallback is important — legacy gifts without a year still count for whatever upcoming occurrence exists, avoiding false "no gift yet" prompts.

## 00:35 EST - Fix regression: Next Best Thing ignores GIVEN gift (td-a76b4e)

### Summary
The td-2de362 fix over-corrected: excluding handled statuses from `looseStmt` caused Amazon-imported gifts with `occasion_id=NULL` and status `given` to be invisible to Today. Roger Bezona's birthday showed "no gift marked bought yet" despite having a GIVEN birthday card.

### What Was Done
- Replaced single `looseStmt` with a two-stage lookup:
  1. `looseActiveStmt` — always safe to claim (active NULL-occasion gifts, year-scoped)
  2. `looseHandledStmt` — only claimed when the person has no one-time occasions in the same year (avoids ambiguity, e.g. Ambree's graduation + birthday both in 2026)
- Added `oneTimeCountStmt` to detect same-year one-time occasions that create attribution ambiguity
- Fixed a better-sqlite3 binding bug: JS numbers bind as REAL, so `CAST(2026 AS TEXT)` produces `'2026.0'`, never matching `SUBSTR(date, 1, 4)`. Passed year as a string parameter instead.

### Files Modified
- `src/lib/server/today.ts` — two-stage loose lookup + oneTimeCount query in `annotateGiftState()`

### Key Technical Insight
better-sqlite3 binds JavaScript numbers as 64-bit REAL, not INTEGER. `CAST(? AS TEXT)` on a bound number produces a decimal string (`'2026.0'`), which silently breaks string comparisons against `SUBSTR()` results. Always pass string-compared values as JS strings.
