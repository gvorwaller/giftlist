# 2026-05-21 Devlog

## td-8360f4 ‚Äî Amazon item exclusion list

### Summary
Keyword-based exclusion list so recurring non-gift Amazon items
(household supplies, subscriptions) stop cluttering the import-review
queue. Per the user's two product calls: filtering is **per item** (a
3-item order with one recurring non-gift still surfaces its gift items),
and the per-item "Exclude" action **prompts the admin to trim the
keyword** to its recurring core before saving so it actually re-matches
future orders. Schema v26 ‚Üí v27. Verified end-to-end live in dev via
claude-in-chrome.

### Design ‚Äî one source of truth, two filter points
Active `exclusion_keywords` rows + a pure `matchExcluded(title, keywords)`
helper, applied in two places:

- **Scan time** (`runAmazonScan`): excluded items are dropped from the
  parsed email before the LLM call + insert, so new rows are clean and
  excluded items cost no matcher calls. A fully-excluded email stages
  `disposition='skipped'` with an `error_message` noting the keyword.
- **Review load**: items are *flagged* (not dropped) so the UI suppresses
  their picker and shows a muted "Excluded by keyword" line ‚Äî this keeps
  `line_item_index` stable and catches rows staged before a keyword
  existed (e.g. right after the admin adds one). The commit path needs no
  change: a suppressed item has no `lineperson_<row>_<idx>` field, so it
  is never turned into a gift.

CRUD mirrors the vendors lookup pattern (soft-delete via `is_archived` +
restore ‚Äî the "restore in case of user error" the task asked for).

### Codex review fix (P1 ‚Äî price/line-item integrity)
Codex (via `codex review`) flagged a data-integrity bug in the original
scan-time filter: dropping excluded items from `parse.items` re-indexed
survivors and could **collapse a multi-item order to a single-item row**.
The commit path then treated the lone survivor as a legacy single item and
used `parsed_price_cents` (the whole **order total**) as its price,
overstating it and losing line-item handling.

Fix: the scan filter no longer mutates `parse.items`/`parse.title`. All
items stay in `parsed_items_json`; the review page hides excluded ones via
the existing load-time flagging, so a partially-excluded order **stays on
the multi-item path** (correct per-line price + stable `line_item_index`).
The only scan-time decision now is "if EVERY item is excluded, skip the
whole email." This also removes the scan-drop-vs-load-flag indexing
inconsistency Codex noted earlier. `npm run check` 0 warnings, 107 tests
green, build clean after the fix.

### Notes / out of scope
