# 2026-05-21 Devlog

## td-8360f4 — Amazon item exclusion list

### Summary
Keyword-based exclusion list so recurring non-gift Amazon items
(household supplies, subscriptions) stop cluttering the import-review
queue. Per the user's two product calls: filtering is **per item** (a
3-item order with one recurring non-gift still surfaces its gift items),
and the per-item "Exclude" action **prompts the admin to trim the
keyword** to its recurring core before saving so it actually re-matches
future orders. Schema v26 → v27. Verified end-to-end live in dev via
claude-in-chrome.

### Design — one source of truth, two filter points
Active `exclusion_keywords` rows + a pure `matchExcluded(title, keywords)`
helper, applied in two places:

- **Scan time** (`runAmazonScan`): excluded items are dropped from the
  parsed email before the LLM call + insert, so new rows are clean and
  excluded items cost no matcher calls. A fully-excluded email stages
  `disposition='skipped'` with an `error_message` noting the keyword.
- **Review load**: items are *flagged* (not dropped) so the UI suppresses
  their picker and shows a muted "Excluded by keyword" line — this keeps
  `line_item_index` stable and catches rows staged before a keyword
  existed (e.g. right after the admin adds one). The commit path needs no
  change: a suppressed item has no `lineperson_<row>_<idx>` field, so it
  is never turned into a gift.

CRUD mirrors the vendors lookup pattern (soft-delete via `is_archived` +
restore — the "restore in case of user error" the task asked for).

### What shipped
- **Migration** `027-exclusion-keywords.sql` — `exclusion_keywords`
  (`keyword`, `match_type` ∈ {contains, exact}, `notes`, `is_archived`,
  timestamps) + index on `is_archived`. No `import_rows` column (per-item
  filtering would make a row-level flag lossy).
- **`src/lib/server/exclusion-keywords.ts`** — list/get/create/update/
  setArchived (create un-archives an archived duplicate instead of
  erroring) + `getActiveExclusionKeywords` + pure `normalizeForMatch` /
  `matchExcluded`. Added `'exclusion_keyword'` to the audit entity union.
- **Scan filter** in `runAmazonScan` — drops excluded items, re-anchors
  `parse.title` to a surviving item, marks fully-excluded rows skipped,
  counts excluded items into the run audit + job summary.
- **Review page** — load flags excluded items/rows (`excludedItemKeys`,
  `excludedRowTitles`); new `?/excludeItem` action creates the keyword and
  redirects with a flash; per-line-item "Exclude this item…" toggles an
  inline trim panel (pre-filled title, contains/exact, Save via
  `formaction`). Single-item rows get the same control and default to
  skip when excluded. Accept-label count subtracts excluded items
  ("create 1 new gift", not 2).
- **Admin CRUD** at `/admin/exclusion-keywords` (add / edit / archive /
  restore) + an "Exclusions" quick-action on `/admin`.

### Codex review fix (P1 — price/line-item integrity)
Codex (via `codex review`) flagged a data-integrity bug in the original
scan-time filter: dropping excluded items from `parse.items` re-indexed
survivors and could **collapse a multi-item order to a single-item row**.
The commit path then treated the lone survivor as a legacy single item and
used `parsed_price_cents` (the whole **order total**) as its price,
overstating it and losing line-item handling.

Fix: the scan filter no longer mutates `parse.items`/`parse.title`. All
items stay in `parsed_items_json`; the review page hides excluded ones via
the existing load-time flagging, so a partially-excluded order **stays on
the multi-item path** (correct per-line price + stable `line_item_index`).
The only scan-time decision now is "if EVERY item is excluded, skip the
whole email." This also removes the scan-drop-vs-load-flag indexing
inconsistency Codex noted earlier. `npm run check` 0 warnings, 107 tests
green, build clean after the fix.

### Notes / out of scope
- Scan-time drops excluded items consistently across an order's emails
  (same keywords), so the Wave 1 fingerprint-first commit dedup still
  links siblings correctly.
- Not retroactive for emails already moved to PROCESSED before a keyword
  was archived (would need Gmail re-staging). Tracking importer is
  untouched (carriers send no item titles).

## Wave 2 — Amazon import automation (Phase 4 + auto-accept + Phase 5)

### Summary
Shipped the full automation arc in one day, in three commits, each through
a `codex review` pass before deploy. North star (Gaylon): make the app do
the daily Amazon-mail work so he isn't a "full-time mail monitor." Roadmap:
`docs/devlog/2026-05-21-amazon-automation-roadmap.md`. Schema v26 → **v28**.

The three tasks share one spine — `detectOverride` ("did the admin's
committed gift differ from the LLM's pick?") with three consumers: cache
invalidation (Phase 4), correction learning (Phase 5), and the auto-accept
undo path. Built once in `src/lib/server/matcher-feedback.ts`.

### Phase 4 — cache spine + invalidation (td-27f36d, commit `95a6551`)
- `matcher-feedback.ts`: `detectOverride`, `invalidateCacheKey`,
  `clearAllCache`, `countCacheRows`. `cache_key` threaded onto
  `LlmMatchVerdict` so it persists with the verdict (invalidation needs no
  shortlist rebuild).
- 4a sweep: `sweepExpiredCache()` folded into the weekly
  `amazon.cleanup_processed` job. 4b: `/admin/system` "Clear LLM cache"
  button. 4c: commit path invalidates the row's cache entry on any
  non-`agree` override.

### Auto-accept — link-only (td-dbaa0c, commit `95a6551`)
- `buildAutoAcceptDecisions`: a row qualifies only when *every* non-excluded
  item is a `high`-confidence match to an existing, non-archived gift —
  never auto-creates. Shipment advances still governed by the abstain.
- Manual "Commit all high-confidence" button on the review page;
  automatic-in-scan behind the `amazon_auto_accept_enabled` app_state toggle
  (`app-state.ts` K/V helper), **default OFF** with a `/admin/system`
  control. The intended path: button → watch hit-rate → flip the toggle.

### Phase 5 — learn from corrections (td-4bfb59, commit `d9bb1df`, schema v28)
- `matcher-corrections.ts` + `matcher_corrections` (mig 028). On an admin
  override/fill-in, `applyOverrideFeedback` appends a correction; the last 5
  feed the already-wired few-shot slot in all three prompt builders. Closes
  the loop: corrections → better confidence → more auto-accept → less manual.

### Codex review (6 findings across 3 rounds, all fixed before ship)
- P1 — scan-filter collapse mis-priced a lone survivor (the exclusion fix
  above, folded into commit `e5236ee`).
- P2 — `detectOverride` only walked `verdict.matches`, missing items the LLM
  left in `unmatched_items`.
- P2 — drive feedback off the **effective committed gift** (a
  `CommittedItem[]` map from `commitMultiItemAccept` + single-item
  created-vs-linked tracking), not the form, so dedup-links register.
- P2 — multi-item corrections logged the row's top-level title, not the
  corrected item's; now per-item.
- P2 — an order's order_placed/shipped/delivered rows appended the same
  correction 2-3×; `appendCorrection` now upserts on a natural key
  (source title + gift + person + action), refreshing recency.
