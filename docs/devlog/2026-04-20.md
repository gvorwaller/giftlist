### Key Technical Insight
- **Svelte 5 `state_referenced_locally` warnings**: reading a reactive prop
  inside `$state(...)` initializers captures a stale snapshot on subsequent
  re-renders. Fix: wrap the initial read in `untrack(() => ...)` from
  `'svelte'`. Legitimate pattern for create-forms where you *want* the
  initial prefill to be a one-shot seed so a failed-action re-render doesn't
  clobber user-typed values.
- **SvelteKit client-bundle poisoning via server-only modules**: pulling a
  pure helper from `$server/...` into a `.svelte` component bundles the
  whole server module (including `getDb` and better-sqlite3) into the client.
  Split pure helpers into `$lib/` and keep DB-touching code in `$server/`;
  have `$server/...` re-export the pure helpers so existing server callers
  don't change. Found via SvelteKit's "If you're only using the import as
  a type..." warning during production build.
- **SvelteKit form actions via curl**: POST without `Content-Type:
  application/x-www-form-urlencoded` returns 415. `curl -X POST -d ""
  -H 'Content-Type: application/x-www-form-urlencoded'` works for status-
  transition testing.
- **Earlier-session bug** (caught Sunday, fixed today conceptually): session
  cookie was originally `SameSite=Strict`, which blocks it from riding
  cross-site redirects back from Google OAuth. Lax is the correct value;
  SvelteKit's origin-header CSRF guard still protects form POSTs.
