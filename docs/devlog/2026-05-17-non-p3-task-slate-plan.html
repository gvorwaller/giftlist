td-1d01e9 P1task — Matcher fix (two phases)

Phase A — heuristic tightening (no API dependency)

File: src/lib/server/gift-matcher.ts

  • Expand stopword list: gift, card, box, set, pack, item, items, the, for, with, and, of, in, on, to, a, an, your, our, my, his, her, their, mr, mrs, ms, dr, jr, sr, christmas, birthday, holiday, graduation, party, day, year.
  • Introduce anchor-token requirement: after stopword/length filtering, the needle must contain at least one token of length ≥ 5 that's not a digit-only string. Match is rejected if no anchor token appears in the haystack.
  • Suppress weak-score outputs whose only overlap is generic stopwords.
Phase A alone kills the Firehouse / graduation match — only overlap is "gift" + "card" (both stopwords); anchor token "firehouse" doesn't appear in the haystack.

Phase B — LLM second-pass at import time

Critical rework per Codex finding #3: LLM calls do NOT happen during page load. They run when the importer stages a row, and results are persisted to import_rows.match_candidates_json (column already exists, migrations/014:54). Page load just renders precomputed results.

Migration

migrations/021-matcher-llm-cache.sql:

CREATE TABLE matcher_llm_cache (
  cache_key  TEXT PRIMARY KEY,
  response   TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_matcher_llm_cache_created ON matcher_llm_cache (created_at);

Code

  • New src/lib/server/matcher-llm.ts. Anthropic SDK, Haiku 4.5. Brief structured-output prompt: needle + candidate list → { best_index | null, reason }. Cache by sha1(needle || sorted_candidate_titles).
  • Hook into amazon-import.ts staging path: when a row is staged, run trigram first; for weak candidates (0.3 ≤ score < 0.6) that survived Phase A's anchor gate, call Haiku and persist the confirmed/rejected result into match_candidates_json.
  • Page load (review/+page.server.ts) reads precomputed candidates from the row — no synchronous LLM call.
  • Add "Re-evaluate matches" admin action on the review page: walks all pending rows of the current run and re-runs Phase A + B. Lets admin retroactively benefit from improved matching on already-staged batches.
  • Failure mode: if Anthropic API unavailable or key absent, persist heuristic-only result. Importer never blocks on API.
  • New env: ANTHROPIC_API_KEY (managed via .env — not via /admin/settings, which is read-only display).
  • Add to .env.example.

Verify

  • Create gift-matcher.test.ts: Firehouse/graduation case (Phase A rejects), Sunmed/gummies true match (Phase A accepts), LLM mock for weak-confirmation (Phase B accepts/rejects).
  • Manual: clear API key, re-stage a run → confirm heuristic-only result persisted; restore key, click "Re-evaluate matches" → confirm cached results returned without re-billing.

td-3d1ee6 P1task — Tracking review queue

Schema — real changes required

Per Codex finding #1, this is NOT a no-op. The disposition CHECK constraint (migrations/014-tracking-import.sql:55-57) only allows pending|accepted|skipped|failed. SQLite CHECK constraints require table recreation to relax.

Migration 019-tracking-review-disposition.sql using the SQLite recreate-table pattern (per CLAUDE.md, migration runner toggles FK off + runs foreign_key_check after):

  1. CREATE TABLE import_rows_new (…) with disposition CHECK IN ('pending','accepted','skipped','failed','review').
  2. Copy all rows from old table.
  3. Drop old table; rename new → import_rows.
  4. Recreate indexes.

Type updates

  • Add console.info logging at each match-clause decision point with row id, order_id, sender_domain, matched_gift_ids (server-side, so logs are useful).

td-d08902 P1task — Multi-recipient shipping (schema-first rework)

Major rescope per Codex finding #2. The current orders table (migrations/018:47) documents itself as "Tracking facts, one shipment per real-world order" — a single tracking_number, carrier, and shipped_at/delivered_at. Partial shipments can't be represented. Need a schema migration before importer changes.

Schema

Migration 020-order-shipments.sql:

CREATE TABLE order_shipments (
  id                          INTEGER PRIMARY KEY AUTOINCREMENT,
  order_pk                    INTEGER NOT NULL
                              REFERENCES orders(id) ON DELETE CASCADE,
  tracking_number             TEXT,
  carrier                     TEXT,
  tracking_provider_id        TEXT,
  tracking_status             TEXT,
  tracking_status_at          TEXT,
  tracking_estimated_delivery TEXT,
  amazon_tracking_url         TEXT,
  shipped_at                  TEXT,
  delivered_at                TEXT,
  source_message_id           TEXT,
  items_json                  TEXT,
  created_at                  TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at                  TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_order_shipments_order_pk ON order_shipments (order_pk);
CREATE INDEX idx_order_shipments_tracking ON order_shipments (tracking_number);
  • Add gifts.shipment_id INTEGER REFERENCES order_shipments(id) ON DELETE SET NULL. A gift belongs to at most one shipment per order.
  • Keep the existing order-level tracking_number/carrier/shipped_at/delivered_at columns on orders as a denormalized "latest/primary shipment" summary for legacy reads. New code reads from order_shipments; legacy reads continue to work without immediate refactor.
  • Backfill: for every existing order with tracking facts, insert a single order_shipments row populated from the order's columns; set every gift under that order to shipment_id = <new_id>.

Importer / lifecycle refactor

File: src/lib/server/jobs/amazon-import.ts

  • Refactor applyLifecycleEvent (lines 726-768) for shipped/delivered events:
    1. Resolve the order via order_id.
    2. If the shipping email body lists item titles (amazon-parser.ts:125-157 extractItemsFromBody), fuzzy-match each shipped item title against the order's sibling gifts using the Phase-A matcher.
    3. Create one order_shipments row capturing tracking#/carrier/timestamps/items_json.
    4. For each matched sibling, set gifts.shipment_id and advance status. Unmatched siblings keep their prior status.
    5. Update the order-level summary columns to reflect the most-recently-shipped shipment (for legacy read compatibility).
  • When the email body doesn't enumerate items (rare), advance ALL siblings under the order and log the case for review.
  • Multi-tracking emails: split by tracking#; one shipment row per tracking#.

Verify

  • Migration: existing prod data still loads after backfill — every order with tracking facts has exactly one shipment row, every gift in that order points to it.
  • New test fixture in amazon-parser.test.ts: 4-item / 3-recipient order, shipping email covering items 1 + 3. Confirm one shipment row created, only siblings 1 and 3 advance.
  • Integration test: 2 shipping emails arriving in sequence for the same multi-item order. Confirm 2 shipment rows, distinct subsets of siblings advanced.
