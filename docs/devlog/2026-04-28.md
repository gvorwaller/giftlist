### Key Technical Insights

- **PM2 env contamination:** `pm2 restart` writes the runtime env (NODE_ENV=production) into `/root/.pm2/dump.pm2`, and that env leaks into a fresh SSH session's shell. A naive `npm ci` then skips devDependencies — vite, svelte-kit, tsx — and the subsequent `npm run build` fails with `vite: not found`. Fix: explicit `NODE_ENV=development npm ci` for the install scope only. Documented at the top of `deploy-to-DO.sh`.

- **Snapshot ordering chicken-and-egg:** First version of the deploy script used better-sqlite3 from node_modules for the pre-deploy DB snapshot. If a previous deploy was OOM-killed mid-`npm ci`, node_modules is in a broken state and the snapshot step fails BEFORE the install step that would repair it. Fixed by switching to the system `sqlite3` CLI (apt install sqlite3) — independent of the app's node_modules.

- **OOM on the 1GB droplet:** `npm ci` peaked above available memory with both PM2 apps running. 1GB swap (swappiness=10) lets the install spill safely without thrashing the running apps. Caught only because the deploy script tested itself end-to-end after being written.

- **commitReviewedRows reuses cleanly for failed-row reassign:** The function doesn't gate on current disposition — calling it with `action: 'accept', assignedPersonId` for a previously-failed row just overwrites the disposition and creates the gift. No special re-process path needed.
