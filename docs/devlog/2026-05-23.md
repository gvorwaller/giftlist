# 2026-05-23 Devlog

## td-2b5c81 — Source URL inconsistent on the gift edit page

### Summary
The edit form's "Source URL" field showed a link for some gifts and an empty
placeholder for others. Reproduced on *Arby's Physical Gift Card – $25*
(Teylor Sonognini): Amazon-sourced, field empty. Diagnosis: **not a save
bug** — a data-model split. `gifts` has two URL columns and the edit form
only reads one. Fix (Option A): the Amazon importer now stores the canonical
order link in `source_url`, plus a one-time backfill for existing rows.
Shipped to prod, schema v28 → **v29**.

### Root cause
| Column | Populated by | Surfaced where |
|---|---|---|
| `source_url` | manual entry + the **Tracking** importer (`https://<sender_domain>/`) | edit form "Source URL" field; not on the detail page |
| `amazon_tracking_url` | the **Amazon** importer (the "Track package" deep-link) | detail page "Track package" link; never in the edit form |

The Amazon import path wrote only `amazon_tracking_url` and left `source_url`
NULL, so Amazon gifts showed an empty field while manual / Tracking-imported
gifts didn't — the "sometimes shows, sometimes doesn't" in the report. For
the Arby's card specifically there's also no `amazon_tracking_url` (gift
cards don't ship), so nothing surfaced either place.

Persistence was fine (`edit/+page.server.ts` → `updateGift` writes
`source_url`), and there's no manager-vs-recipient rendering difference — the
field is the same component for both roles. The gap was purely *which column
each import path populated*.

### What shipped
- **`amazonOrderUrl(orderId)`** helper in `jobs/amazon-import.ts` —
  `https://www.amazon.com/gp/css/order-details?orderID=<order_id>`,
  `encodeURIComponent`'d (no-op for the digit/hyphen order-id format, hardens
  against a malformed parse), null-safe.
- Wired into **both** Amazon `createGift` paths (single-item + multi-item
  commit), so new Amazon gifts populate `source_url`. No frontend change —
  the edit form already binds `data.gift.source_url`.
- **Migration `029-gift-source-url-amazon-backfill.sql`** — one-time UPDATE
  for existing Amazon gifts. Scope guards: `source_url IS NULL` (never
  clobber a manual URL), `order_id` present and non-blank, vendor named
  `Amazon`.
- 3 new unit tests for `amazonOrderUrl` (standard id, null/empty, malformed).

### Decision — why Option A
Considered: (B) show `amazon_tracking_url` as its own read-only line on the
edit page, (C) relabel Source URL as manual-only. Picked A because it matches
the mental model ("the Amazon link"), fills the existing field with a
genuinely useful link (opens the order), and is the smallest change. The
order link is also semantically the right thing for "source" — the shipment
tracker stays where it belongs on the detail page.

### Verification
- `npm run check` 0 warnings; `npm test` **137 passing** (was 134, +3);
  `npm run build` clean.
- Migration applies on a fresh DB (test harness reaches v29).
- Backfill UPDATE validated against a throwaway SQLite DB: backfilled the one
  eligible Amazon row + touched `updated_at`; left manual-URL, non-Amazon,
  no-order-id, and blank-order-id rows untouched.
- Deployed `cdf4da5` via `deploy-to-DO.sh` (pre-deploy prod snapshot). Prod
  `/healthz` reports `schema_version: 29`; boot logged `[boot] schema at
  version 29`, so the backfill ran against prod.

### Notes
- Backfill is keyed on the Amazon **vendor name**, so any gift manually
  attributed to Amazon with an order id also gets the link — intended.
- Tracking importer untouched (it already sets `source_url`).

## Follow-up — Source URL as a tappable button on the edit form

### Summary
Once the backfill landed, the edit form showed the order URL as raw text in
the input. On mobile that's useless — to use it you'd copy/paste into a
browser, at which point opening the Amazon app directly is just as fast. Made
the field render a one-tap **"Open link ↗"** button (opens in a new tab) when
a URL exists, with a small **Edit** toggle that swaps back to the text input.
Empty URLs still show the plain input. Shipped to prod (commit `51710f7`, no
schema change — still v29).

### What shipped
- `app/gifts/[id]/edit/+page.svelte`: `source_url` field is now conditional —
  `{#if sourceUrl && !editingSourceUrl}` renders the `Open link ↗` anchor
  (`target="_blank"`, `rel="noopener noreferrer"`) + an Edit button, with a
  hidden `source_url` input so the value still posts on save. Otherwise the
  editable text input (manual-entry / correction path) is shown.
- New `editingSourceUrl` state, default `false`; flips to `true` when Edit is
  tapped. After save the page reloads and it resets.
- Button styling mirrors the detail page's "Open Amazon tracking ↗" affordance
  (green-soft fill, full `--tap-target` height for the 48px touch minimum).

### Why this shape (vs. detail-page button / both pages)
Asked about placement; the user clarified the ask was specifically the edit
form's field — replace the dead URL text with something tappable, not add a
button elsewhere. Kept editability because manually-added gifts still need to
enter/fix the URL, so a hard input→link swap would've been a regression. The
Edit toggle is the compromise: link by default (the common, auto-filled case),
input on demand.

### Verification
- `npm run check` 0 warnings; `npm test` 137 passing; `npm run build` clean.
- Deployed `51710f7` via `deploy-to-DO.sh` (pre-deploy prod snapshot). Public
  `/healthz` ok, `schema_version: 29`.

## Follow-up #2 — open Source URL via a redirect hop (iOS Universal Links)

### Summary
The new "Open link ↗" button worked, but tapping it on iPhone opened the
**Amazon app** and landed on "Your Orders" — not the specific order. Pasting
the same URL into Safari's address bar *did* show the order. Root cause is iOS
**Universal Links**, not our URL. Fixed by routing the tap through a redirect
on our own domain. Shipped to prod (commit `a02a2a2`, still schema v29).

### Root cause
When you tap an `amazon.com` link from inside another app or a Safari page,
iOS checks whether the Amazon app has claimed that domain/path and, if so,
hands the URL to the app. The app's deep-link router doesn't recognize the
`/gp/css/order-details` *web* path as a specific order screen, so it falls
back to "Your Orders." A copy/paste into the address bar is a **typed**
navigation, which iOS never routes to an app — so it loads the real web page
and renders the order. Same URL, different entry path.

### The fix — a redirect iOS won't intercept
iOS does **not** apply Universal-Link interception to server **redirects**
(the well-known reason link-shorteners "break" universal links). So:

- **New `GET /app/gifts/[id]/open-source`**
  (`src/routes/app/gifts/[id]/open-source/+server.ts`) — requires login,
  re-uses the gift-detail self-package privacy guard (td-68804e: 404 a
  foreign/orphaned self-gift), validates `source_url` is a real `http(s)` URL
  (else 303 back to the gift page so a blank/odd value can't emit a strange
  `Location`), then `302`s to `source_url`.
- The edit form's "Open link ↗" button now points at that route
  (`href="/app/gifts/{id}/open-source"`) instead of the raw URL.

Because `gifts.gaylon.photos` isn't an app-claimed domain, the tap stays in
the browser; Safari then follows the 302 to amazon.com **without** opening the
app — i.e. the order page, exactly like copy/paste. No open-redirect risk: the
destination comes from our own DB row, never the query string.

- `npm run check` 0 warnings; `npm run build` clean.
- Deployed `a02a2a2` via `deploy-to-DO.sh`. Public `/healthz` ok,
  `schema_version: 29`.

## Follow-up #3 — the redirect hop didn't work; "Open link" lands on gift detail

### Outcome
The redirect hop (#2) did **not** beat Universal Links on the user's iPhone:
current iOS follows the 302 and *still* hands the `amazon.com` destination to
the Amazon app (the user saw a blank page in our app, then the app's "Your
Orders"). Reverted it (commit `024a499`). Confirmed there's **no web-side way**
to force a tapped `www.amazon.com/gp/css/order-details` link into the browser
once the app claims that host — copy/paste works only because typed URLs are
never routed to apps.

### Why the tracking link *does* work (and the order link can't)
History dig: nothing was removed. The only external Amazon link the UI has
ever rendered is **"Open Amazon tracking ↗"** on the gift detail page (since
`td-b221ae`). It opens in the browser because it targets **`track.amazon.com`**
— a host the Amazon app does **not** register with iOS. The order link targets
**`www.amazon.com`**, which the app *does* register, so iOS intercepts it and
(ignoring the order id) shows Your Orders.

### What shipped (per the user's "both links" choice)
- **Gift detail hero** (`app/gifts/[id]/+page.svelte`): an **"Open link ↗"**
  button now renders whenever `source_url` exists, alongside the existing
  track.amazon.com tracking link. Desktop / no-app: opens the order page.
  iPhone-with-app: opens the Amazon app (an accepted tradeoff — the tracking
  link is the one that stays in-browser).
- **Edit form**: button reverted to a plain `href={source_url}` (dropped the
  redirect indirection + its blank-page hop).
- Deleted `app/gifts/[id]/open-source/+server.ts`.

### Verification
- `npm run check` 0 warnings; `npm test` 137 passing; `npm run build` clean.
- Deployed `024a499` via `deploy-to-DO.sh`. Public `/healthz` ok,
  `schema_version: 29`.
