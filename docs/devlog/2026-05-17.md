### What shipped

**td-c12570 (P1 bug) — Amazon-review buttons unresponsive.** Single-line
CSS fix: `z-index: 30` on the sticky `.actions` bar in
`src/routes/admin/imports/amazon/review/+page.svelte`. The BottomNav
(fixed, `z-index: 20`) was eating taps when the action bar scrolled
underneath it. Verified the same pattern existed on the tracking review
page and patched both. Codex correctly flagged that the planned
`use:enhance` rewrite was scope creep — narrowed to CSS-only.

**td-1d01e9 Phase A (P1) — Matcher heuristic tightening.** Expanded the
stopword list (`gift`, `card`, `box`, `christmas`, `birthday`,
`graduation`, etc. — anything appearing in ~30%+ of titles) and added
an anchor-token gate: the candidate's title must contain at least one
non-stopword token of length ≥ 5 (non-digit), AND that anchor must
appear in the email haystack, before scoring kicks in. Refactored
`matchGiftByTitle` to delegate to a new pure `scoreGiftCandidates()` for
unit testability. Seven new tests in `gift-matcher.test.ts`, including
the user's screenshot regression (Firehouse-gift-card weakly matching a
2026 Graduation Card row at 33% on `card` overlap alone).

**td-1d01e9 Phase B (P1) — LLM matcher second-pass.** Codex caught a
critical design flaw in v1 of the plan (LLM call in page-load path,
blocking on Anthropic API). Rescoped to import-time / admin-triggered:
new `migrations/021-matcher-llm-cache.sql` (sha1 of needle + sorted
candidate titles), new `src/lib/server/matcher-llm.ts` using raw `fetch()`
against the Anthropic API (no new SDK dependency — `@anthropic-ai/sdk`
not added). `matchGiftByTitle` consults the cache synchronously for
weak matches (one indexed lookup), never blocks on the API. The
"Re-evaluate weak matches with AI" admin button on the Amazon-review
page calls `reevaluateMatchesForRun` which walks pending rows, calls
Haiku 4.5 for any not yet cached, and writes results. UI surfaces
"AI confirmed" / "AI rejected" / "AI pick" badges with the LLM's
one-sentence reason. Graceful no-op when `ANTHROPIC_API_KEY` is unset.

**td-3d1ee6 (P1) — Tracking review queue.** End-to-end review surface
for the silent-orphan-self-package bug. New `migrations/019` widens the
`import_rows.disposition` CHECK constraint to include `'review'`
(recreate-table pattern). `acceptTrackingRow` and
`acceptOrderConfirmationRow` in `tracking-import.ts` now check for ANY
gift with the parsed `order_id` before fabricating a self-package; if
candidates exist but sender/vendor evidence is ambiguous, the row is
routed to `disposition='review'` with the candidate list serialized to
`match_candidates_json`. New `resolveTrackingReview()` function with
three actions (`attach` to chosen candidate / `self_package` to bypass /
`skip`). The existing `/admin/imports/tracking/review/` page was
extended (not replaced — Codex finding #6) to surface review rows in a
new amber-styled section above pending, with per-row candidate cards
and three resolve actions per row.

**td-d08902 (P1) — `order_shipments` schema + lifecycle refactor.** The
big one. Codex correctly flagged that v1 of the plan would create
inconsistent state under the existing one-shipment-per-order model.
`migrations/020` adds `order_shipments` (N per order) + `gifts.shipment_id`,
backfilling one shipment per existing order from the order-level columns
(orders' `tracking_number` / `carrier` / `shipped_at` columns kept as a
denormalized "primary shipment" summary for legacy read compatibility).
`applyLifecycleEvent` in `amazon-import.ts` now creates per-shipment
rows for `shipped`/`delivered` events and uses the Phase-A matcher
(`matchSiblingsToShipment` helper in `orders.ts`) to advance ONLY the
sibling gifts whose titles match items listed in this shipment's email
body. Falls back to advancing all siblings (with a warn log) when the
email body doesn't enumerate items. Four new tests in `orders.test.ts`
cover the 2-of-4 partial-shipment case plus edge cases.

**td-77a119 (P2) — Searchable `<PersonPicker>` combobox.** New

   to CSS-only z-index.

   added `archived_at` column.
6. Tracking review route already existed (extend, don't create);
   `match_candidates_json` already existed; archived gifts already
   surfaced in "Past gifts" per-person history.
7. Migration numbering was wrong — repo was at v18, plan said v16.

The plan rewrite was significant. The slate ships better as a result.

### Key technical insight
Codex's "schema-first design" guidance on `td-d08902` was the single
most valuable piece of review feedback. The original lifecycle-only
plan would have patched `applyLifecycleEvent` to advance subsets of
siblings, but with no place to store which tracking number applied to
which subset, the on-disk state would have ended up inconsistent —
`orders.tracking_number` reflects whichever shipment email arrived
last, but gifts would have been individually advanced based on whatever
fuzzy match fired at the time. After two boxes had landed, the
"primary" tracking on the order would point at one of them and the
other would be lost. The `order_shipments` table is a small schema
addition that makes the per-shipment state actually addressable, and
the legacy denormalized columns on `orders` mean nothing else in the
codebase had to change at the same time.
