as the primary Amazon-import matcher, shipment-decider with abstain on
high-stakes shipment matching, and commit-time dedup with content

### What shipped

**New modules:**
- `src/lib/server/llm-matcher.ts` — Opus 4.7 via raw `fetch()` against
  the Anthropic Messages API, tool_use-enforced structured output,
  versioned cache key. Public surface: `llmMatchImportRow` and
  `llmMatchShipment`. Decision contract is
  `{matches: [{itemIndex, giftId, confidence: 'high'|'medium'|'low',
  reason}], unmatched_items[], safe_to_apply, summary, model,
  prompt_version, created_at}`. Shipment-mode forces
  `safe_to_apply: false` whenever any match is `low` or
  `unmatched_items` is non-empty.
- `src/lib/server/shipment-decider.ts` — owns the safety policy. If the
  heuristic matches every shipped item against a sibling cleanly, use
  it. Otherwise call `llmMatchShipment`. Honor `safe_to_apply`; on
  abstain, create the shipment row anyway (tracking data is real) but
  return `{kind: 'abstain', reason}` and the caller does NOT advance
  any siblings.
- `src/lib/server/test-harness.ts` — tempfile-backed SQLite for tests
  with all migrations applied + factories for users / people / gifts /
  import runs / import rows. Lets the fixture corpus exercise the real
  commit pipeline with the LLM mocked.

**Modules rewritten or significantly refactored:**
- `src/lib/server/gift-matcher.ts` — purpose changed from pass/fail
  gate (the prior STRONG/WEAK/NONE confidence bands) to
  candidate-shortlist ranker. `rankCandidatesForImport(needleTitle,
  recipientHintPersonId, limit)` returns top-N enriched
  `MatcherCandidate` rows (joined with people + occasion) with
  recipient-hint priority. `siblingsAsCandidates(orderPk)` is the
  shipment-mode equivalent. The old `matcher-llm.ts` (Haiku
  weak-match-only second opinion) is deleted.
- `src/lib/server/jobs/amazon-import.ts` — the big one. In the scan
  path, each pending row now gets an Opus verdict dispatched in
  parallel and persisted on the row's `llm_verdict_json`. The commit
  pipeline picks up a `ShipmentAdvancePlan` per shipped/delivered row
  asynchronously BEFORE the sync transaction (LLM calls don't belong
  inside `db.transaction`). `commitMultiItemAccept` is rewritten with
  deterministic dedup: fingerprint-first, line_item_index as fallback,
  recipient-conflict refusal, transactional wrap.
  `applyLifecycleEvent` consumes the pre-flight plan; on abstain it
  creates the shipment row but no sibling advances.
  `reevaluateImportRowsForRun` replaces the old
  `reevaluateMatchesForRun`.

### Phase 3 dedup behavior (post-test refinement)

First pass had two bugs caught during dev testing in browser:

1. **Match order was wrong**: tried `(order_pk, line_item_index)`
   exact first. But Amazon emails commonly reorder items between
   `order_placed` and `shipped/delivered` — line_item_index 0 in the
   shipped email is rarely the same item as line_item_index 0 in the
   order_placed email. Triggered false recipient-conflicts.
   **Fix:** fingerprint-first, line_item_index only when fingerprints
   don't match.

2. **Fingerprint was over-keyed**: original `sha1(title + price)`
   produced false-misses because the parser populates the per-line
   price on some Amazon emails (`order_placed`) but not others
   (`shipped`). Same product, two different fingerprints.
   **Fix:** title-only fingerprint. Amazon's titles for fixed-amount
   items include the denomination ("MasterCard ... $100"), so titles
   alone distinguish them without needing the unreliable parsed price.

3. **Link branch rewrote `line_item_index`** which collided with the
   partial unique index when Amazon reordered. **Fix:** preserve the
   first-set `line_item_index` on linked gifts; only stamp it when the
   existing value is null (e.g. when an "idea" gift gets linked to a
   multi-item order for the first time).

### Codex review followup (the gap list)

After the initial Wave 1 ship, audited honestly against the plan and
the Codex review and closed the eight gaps that remained:

1. **Fixture regression corpus** — Codex review #7 called for
   sanitized real-email fixtures with golden expectations as "the
   primary regression suite, not just local smoke." Built
   `src/lib/server/jobs/amazon-import-fixtures.test.ts` with 8 cases:
   order_placed multi-item, shipped_partial, shipped_no_enumeration
   (LLM abstain path), shipped_reordered_items (fingerprint dedup
   under Amazon reorder), duplicate_titles within one commit,
   duplicate_titles across separate commits (dedup correctly links),
   same_recipient_multi_items, conflicting_override (recipient
   mismatch surfaces). Each runs the real commit pipeline against a
   fresh tempfile DB with `llmMatchImportRow` / `llmMatchShipment`
   mocked. 82 unit tests total now pass.
2. **Abstain UX surfacing** — the review page reads new `?abstained=N`
   query param and shows an amber flash banner. Accepted-but-held
   rows get a "held" tag + amber row background + their
   `error_message` displayed inline. Admin no longer has to dig.
3. **Counter split** — `CommitResult` now exposes `giftsCreated`,
   `giftsLinked`, `siblingsAdvanced`, `rowsAbstained` separately. The
   redirect URL carries them as `?created=&linked=&advanced=&abstained=&…`
   and the flash banner shows each metric individually. The misleading
   "gifts=N" that conflated created with linked is gone.
4. **Body fallback** — `parsed_body_excerpt` (≤4000 chars) captured at
   scan time when the items[] extractor came up empty. The LLM matcher
   uses it as fallback context. Cache key absorbs a body-hash so
   different excerpts produce different cache entries.
5. **`?gifts=N` cosmetic bug** — subsumed by #3.
6. **Re-evaluate button label** — relabeled in Phase 6 below.
7. **Docs** — this file + CLAUDE.md schema/architecture refresh +
   README stack + features + key-files update.
8. **B4/B5 single-item shipped + heuristic-uncertain abstain
   fallback** — covered by fixture cases #2 (shipped_partial) and #3
   (shipped_no_enumeration → abstain).

### Migration 025 forensic record

The unique-index migration couldn't install against prod data as-is —
the very bug Wave 1 fixes had been creating duplicate gifts under
shared `(order_pk, line_item_index)` slots for weeks. Five duplicate
clusters identified on prod snapshot:

```
order_pk=1, line_item_index=0: 2 gifts
order_pk=1, line_item_index=2: 2 gifts
order_pk=4, line_item_index=0: 4 gifts
order_pk=4, line_item_index=1: 2 gifts
order_pk=4, line_item_index=2: 2 gifts
```

Per the user's choice, migration 025 auto-archives all but the
lowest-id gift in each cluster, persists the archived gift IDs +
title + person + status to
`app_state.wave1_migration_025_archived_gifts` as JSON for forensics,
then creates the index. 7 prod gifts will be archived on first boot
post-deploy. Admin should review the forensic JSON and Restore any
"wrong-kept" ones via the existing `/admin/system/archived` browser.

### Dev test loop

`scripts/dev-wave1-iteration.sh --restore` cleanly rolls back local DB
to the pre-test snapshot, relabels Gmail messages from
`Giftlist/Amazon/Processed` back to `Giftlist/Amazon/Inbox`, and
relaunches the dev server. Tested end-to-end with 5 seeded open gifts
+ 50 real Amazon emails relabeled to inbox:

- 30/43 pending rows got Opus verdicts persisted in
  `import_rows.llm_verdict_json` during the scan (the other 13 had
  empty heuristic candidate pools — correct skip).
- Cache populated 29 entries under mode='import' (one cache hit
  collapsed across duplicate row contexts — versioned key working).
- LLM reasoning samples observed:
  - "MasterCard gift cards are generic stored-value cards; no
    candidate matches. The 'Graduation card 2026' is a paper greeting
    card, not a prepaid gift card." (correctly rejected $100
    MasterCard → graduation card)
  - "This is a graduation card specifically for a grandson; the only
    graduation card candidate is for Madison Moore (female recipient
    per item 1 context), so no candidate fits." (used cross-item
    context to reject)
- `Accept → advance N existing gifts` label flipped correctly from
  "create N new" to "advance N existing" after the order_placed was
  committed. Phase 3 dedup-aware UI.
- Committed an `order_placed` row (5 line items, 5 new gifts at
  indexes 0-4), then a `shipped` row for 2 of those items. Final DB
  state: exactly 5 gifts under the order (zero duplicates), 2
  advanced to `shipped` with `shipment_id` attached, 3 stayed at
  `ordered` (correctly NOT in this shipment). One `order_shipments`
  row created with tracking info.
