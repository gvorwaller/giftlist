# 2026-05-26 Devlog

## 11:20 EST - Fix Amazon matching creating duplicates instead of linking (td-7e49bc)

### Summary
The LLM matcher's candidate pool was too narrow, filtering out any gift past `planned` status. Manually-created gifts that the user had already advanced to ordered/shipped were invisible to the matcher, causing the Amazon import to create duplicates instead of linking.

### What Was Done
- Diagnosed the issue using prod DB: Marcus had gifts 50 (Osprey, title was a URL) and 51 (Godiva, abbreviated title), both manually advanced to `shipped` before the Amazon order_placed email arrived
- Traced the failure through the full pipeline: `rankCandidatesForItems` → `buildAndCallMatcher` → commit path
- Root cause: `OPEN_GIFTS_QUERY` in `gift-matcher.ts` filtered `g.status IN ('idea', 'planned')`, so the LLM was never called (zero candidates → null verdict → no `llm_verdict_json`)
- At commit time, `commitMultiItemAccept` couldn't dedup either: no `order_pk` on manual gifts, no fingerprint match (URL title vs product name)
- Fix: changed status filter from `IN ('idea', 'planned')` to `NOT IN ('given', 'returned')` so all active statuses are eligible
- Also updated the `siblingsAsCandidates` string-replace that mirrors the WHERE clause
- All 134 tests pass, zero type errors
- Deployed to production

### Files Modified
- `src/lib/server/gift-matcher.ts` — widened `OPEN_GIFTS_QUERY` status filter + matching `siblingsAsCandidates` replace string

### Key Technical Insight
The `order_id IS NULL` filter in the candidate query is correct and stays: gifts already claimed by a specific Amazon order are matched via `resolveOrCreateGift`'s direct order_id lookup, not the LLM. The status filter was the sole bottleneck. The LLM (Opus 4.7) is smart enough to match a URL-titled gift to its product name or an abbreviated title to a full Amazon listing — the problem was it never got the chance to see them.

### Next Steps
- Monitor next Amazon scan to confirm the fix works end-to-end with real emails
- Consider whether the gift-matcher test suite should cover the "manually advanced gift" scenario explicitly
