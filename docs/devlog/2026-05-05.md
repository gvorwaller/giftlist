# 2026-05-05 Devlog

## 14:10 EDT - Filter self-orders out of gift-manager views (td-24eac3)

### Summary
Implemented the scope-expansion filter so personal (non-gift) orders can ride
the same database without polluting the gift-manager flows. `people.is_self`
flag added; `/app/today`, `/app/people`, and reminder digests now hide self-
people while `/app/packages` continues to show everything in flight.

### What Was Done
- **Schema (migration 012-people-is-self.sql)**: added `is_self INTEGER NOT NULL DEFAULT 0 CHECK (is_self IN (0, 1))` to `people` plus an index. All existing rows default to `is_self=0`. Verified by applying to a copy of the dev DB and inspecting the schema.
- **Type + data layer (`src/lib/server/types.ts`, `people.ts`)**:
  - `Person.is_self: 0 | 1` field added.
  - `PersonUpsertInput.is_self?: boolean` so admin forms can write the flag.
  - `listPeople({ includeSelf })` defaults to `false` (manager views) and `true` only where the admin opts in. Implemented by appending `p.is_self = 0` to the WHERE clause when `includeSelf` is omitted.
  - `createPerson` / `updatePerson` honor the flag; updatePerson preserves the prior value when the input field is undefined (so non-admin code paths can't accidentally clear the flag).
- **Manager-view filters**:
  - `today.ts loadUpcomingWindow()` JOIN now requires `p.is_self = 0`.
  - `today.ts` packagesOnTheWay query joins `people` and excludes self.
  - `jobs/reminders.ts collectUpcoming()` and `collectPackages()` both gain the same filter — keeps the digest focused on actual gifts.
  - `admin-home.ts countIncompletePeople()` / `listIncompletePeople()` also exclude self so the admin "incomplete people" warning doesn't fire just because Gaylon's self-row has no birthday.
- **Admin UI**:
  - `/admin/people` listing now passes `includeSelf: true`, shows a "Self" badge + "(me)" inline tag + amber subline ("Personal orders — hidden from manager views").
  - `/admin/people/new` and `/admin/people/[id]` got an `is_self` checkbox with descriptive helper text.
- **Manager UI**:
  - `/app/people` keeps the default filter, so self-people don't appear there.
  - `/app/gifts/new` includes self-people in the dropdown with a `(me)` suffix; the load function skips occasion auto-pick when the active person is self (occasion stays optional, no auto-default to nearest birthday).
- **Verification**: `npm run check` clean (0 errors / 0 warnings). Boot smoke test confirmed migration 012 applied successfully on the dev DB (`schema_version=12`, all 49 people defaulted to `is_self=0`).

### Key Technical Insight
Chose option (a) `is_self` boolean over option (b) reserved-row pattern. Three reasons: (1) keeps the person model uniform — self-orders still have a real person row, just with a flag; (2) the filter generalizes (`WHERE is_self = 0`) instead of relying on a magic ID lookup that would need to be threaded through every query; (3) supports multiple selves later without code changes (e.g., spouse's orders later become a second self-row).

The "preserve prior value when input.is_self is undefined" detail in `updatePerson` is intentional — non-admin code paths (e.g., a future contacts-import refresh) shouldn't accidentally toggle the flag just because they don't pass it. Admin forms always pass it (checkbox state).

## 14:28 EDT - Codex review patches (td-24eac3)

### Summary
Codex review of the uncommitted td-24eac3 diff caught three real leak/foot-gun
paths that the AC didn't enumerate. Patched all three.

### Findings + Fixes
1. **Self-person leak through `recently_viewed` + `/app/people/[id]`** — the manager-facing person detail route only blocked archived rows, so a self-person was reachable via direct URL, which then wrote to `recently_viewed`, which `/app/today` rendered without filtering.
   - `src/routes/app/people/[id]/+page.server.ts`: now also 404s when `person.is_self === 1`.
   - `src/lib/server/today.ts`: replaced the `listRecent` call with an inline query that filters out self-person rows AND gift rows whose person is_self. Also dropped the now-unused `listRecent` import.
2. **Editing a self-order from `/app/packages` was broken** — `/app/packages` (correctly) shows self-orders, but `/app/gifts/[id]/edit` built its people dropdown from `listPeople` defaults (excluded self), so the form's required `<select>` had no option matching the gift's person_id and the page failed validation.
   - `src/routes/app/gifts/[id]/edit/+page.server.ts`: pass `includeSelf: true` to listPeople; if the gift's person is_self, return an empty `personOccasions` list.
   - `src/routes/app/gifts/[id]/edit/+page.svelte`: append " (me)" suffix to self options, matching the gift-new form.
   - Bonus: `src/routes/app/gifts/[id]/+page.svelte`: don't render person-detail links for self-orders (they would 404 now); render plain text instead.
3. **Google Contacts import could silently adopt a self-row by full_name** — the preview + commit paths matched existing people via `full_name` only, so a self-person whose full_name happened to match a Google contact would be adopted and have a birthday assigned to it.
   - `src/lib/server/contacts-import.ts`: both `byFullName` queries gain `AND is_self = 0`.

### Codex non-issues (cleared)
- `updatePerson` "preserve prior is_self when undefined" logic — verified not bypassed; contacts-import uses direct SQL for adoption/insert paths, never `updatePerson`.
- `/admin/audit` filtering — audit listing reads from `audit_log` directly with no `people` join, so it's not affected.
- Migration interaction with the partial unique index on `google_resource_name` — additive only, no conflict.

### Key Technical Insight
The `recently_viewed` leak path was the most subtle one — even with `/app/today`, `/app/people`, and the reminder digest all filtered correctly, a self-person could still appear on the Today screen because the recent-history rendering trusted the table at face value. Lesson: any user-facing list that joins to `people` (even transitively, via a foreign key like `recently_viewed.entity_id`) needs the same `is_self = 0` filter applied at render time, not just at write time. The "filter at read, defense in depth" approach handles the case where rows were written before is_self was introduced or before the filter was added.
