## 09:30 EDT — Phase 1 remediation: three surgical fixes shipped together

### Summary
Closed three td issues in one batch — all small, well-scoped P1/P2 fixes.
Phase 2 (multi-recipient order split) is queued as a separate, larger
schema-migration body of work.

### What Changed

**td-68e82a (P1) — Birthday off-by-one in Next Up.**
`nextOccurrenceDate()` in `src/lib/server/occasions.ts` constructed
`new Date(year, month-1, day)` — a *local-timezone* midnight. Production
runs PM2 on a DigitalOcean droplet that's typically UTC. The resulting
"Dec 7 00:00 UTC" date serialized over the wire arrived in the US Eastern
browser as "Sunday, December 6". The day-of-week shift was the smoking
gun: it pointed at TZ slip, not date arithmetic. Math.round vs Math.floor
was a red herring.

Fix: anchor occurrence dates at **noon UTC**. Noon UTC is the same
calendar date in every timezone from UTC−11 through UTC+11, so JSON
round-trips preserve the intended day-of-month. Exported a small helper
`todayMidnightUTC()` and converged `today.ts`, `people.ts`, and
`reminders.ts` on it. New unit tests in `occasions.test.ts` lock in the
behavior under multiple US timezones.

**td-0c8de5 (P1) — Reminder "needs attention" conflicts with bought gifts.**
The digest's `HANDLED_STATUSES` constant in
`src/lib/server/jobs/reminders.ts` was `{given, returned}` — too narrow.
That meant Ambree's `$50 gift card` (status: `ordered`) was being routed
into the *needsAttention* bucket and rendered as "No gift is marked
bought yet" — while the very same digest's "Packages on the way" section
listed the same gift. Broadened to all post-`idea` statuses
(`planned, ordered, shipped, delivered, wrapped, given, returned`).
"Coming up" still surfaces every committed occasion through delivery,
unchanged. Updated the explanatory comment to spell out the digest's
semantic split.

**td-8de343 (P2) — Gift status badge in Coming Up Soon.**
Added `bestGiftStatus: GiftStatus | null` to `UpcomingOccasion` and a
single CASE-ordered query (`annotateGiftState`) that fetches the most-
progressed non-archived gift per `(person, occasion)`. Rendered as a
text-labeled pill in `/app/today` with three accessible tone variants
reusing existing design tokens — no new colors, AAA contrast preserved.
Existing `hasHandledGift` semantics ({given, returned}) intact so
Today's "stay until given" filter is unchanged.

## 11:00 EDT — Phase 2 remediation: multi-recipient order split

### Summary
Closed td-3e9ae2 and its design-notes companion td-6c189e in one
implementation. Schema migrated to v18 with a new `orders` table
sitting 1:N over `gifts`. Amazon imports now parse every line item
from a multi-item email, the review UI renders a per-line recipient
picker, and the commit path creates one order + N gifts. The single-
item path is unchanged in behavior — it just additionally writes a
parent orders row going forward.

### What Changed

**Schema — migrations/018-orders-table.sql.**
- New `orders` table owns order-level facts: order_id (unique),
  vendor/shipper FKs, total cents, tracking number/carrier/provider
  id/Amazon URL, lifecycle timestamps, source message id.
- `gifts` gains `order_pk` (FK) + `line_item_index` (0..N-1 within an
  order). Existing tracking columns on `gifts` stay denormalized for
  one release for back-compat with all reader code.
- `import_rows` gains `parsed_items_json` (JSON array of
  `{title, priceCents, quantity}`). Null for legacy rows.
- Backfill: for every distinct `gifts.order_id`, INSERT one synthetic
  `orders` row and stamp `gifts.order_pk` + `line_item_index = 0`.
  Tested against a copy of the production DB via the real `migrate.ts`
  runner — schema bumped to 18, no foreign-key violations.

**Parser — src/lib/server/amazon-parser.ts.**
- New `ParsedAmazonItem = { title, priceCents, quantity }` and
  `parseAmazonEmail().items: ParsedAmazonItem[]`.
- New `extractItemsFromBody()` does a two-pass walk: pass 1 finds
  every `* TITLE / Quantity: N` header and its byte range; pass 2
  pulls the first `$X.XX` inside that range as the line-item price.
  Tolerates items without an inline price (priceCents = null).
  Backed by 4 new tests including a faithful reproducer of order
  #113-2234245-9301002's four line items.

**Persistence — src/lib/server/jobs/amazon-import.ts.**
- `insertRowStmt` now includes `parsed_items_json`.
- New `orders.ts` helper module with `upsertOrderByOrderId()` (fill-
  only via COALESCE; safe to call across order_placed → shipped →
  delivered).
- New `ensureOrderForRow()` upserts on the row's `parsed_order_id`,
  called from `resolveOrCreateGift()` so every Amazon import (single-
  or multi-item) gets a parent order row going forward.
- New `commitMultiItemAccept()` creates 1 order + N gifts when the
  admin specified per-line recipients in the review UI.
- `applyLifecycleEvent()` now walks the `order_pk` sibling set —
  shipped/delivered emails advance ALL gifts under the order, not just
  one. Falls back to single-gift behavior when no order_pk exists
  (legacy gifts).

**Commit input — `CommitRowInput.lineItems`.**
- Optional array of `{lineItemIndex, assignedPersonId, assignedGiftId?}`.
- When present, the loop routes to the multi-item helper and skips the
  single-gift legacy path for that decision.

**Review UI — /admin/imports/amazon/review.**
- Loader parses `parsed_items_json` into `rowItems[r.id]` and also runs
  per-line-item weak match (`lineItemMatches['<rowId>:<idx>']`).
- Pending rows with 2+ line items render a `.line-items` fieldset
  containing one `<li>` per item: title + price + recipient picker
  + per-item weak-match candidates. A "Same recipient for all N"
  shortcut bulk-updates every picker via a small client-side helper.
- For multi-item rows, the legacy single-row gift-link block and
  "Or assign to person" picker are hidden (they'd be misleading —
  each item gets its own).
- `parseDecisions()` gathers `lineperson_<row>_<idx>` and
  `linegift_<row>_<idx>` form fields into the new
  `CommitRowInput.lineItems` array.

**Backfill route — `?/reSplit` on /admin/imports/amazon.**
- New `reImportOrderById()` re-fetches the original Gmail message,
  re-parses with the new multi-item logic, refreshes
  `parsed_items_json` + `parsed_title` + `parsed_price_cents` on the
  existing order_placed import row, archives the collapsed gift with
  an audit-log breadcrumb (restorable from `/app/gifts/[id]`), and
  flips the row's disposition back to `pending` so the admin can
  assign N recipients from the review UI.
- Wired up as a `?/reSplit` form action on the Amazon imports
  landing page with an order-ID input matching the `XXX-XXXXXXX-XXXXXXX`
  pattern. Idempotent — safe to re-run.

### Deferred / follow-up

  `tracking.ts` to key on `orderPk`) can ride a follow-up commit.
- **Drop legacy `gifts.tracking_*` columns.** Pinned to a one-release
  window for safe rollback; a future migration can drop them once new
  code has soaked.
